# JAVA

# 基本概念

JDK、JRE、JVM 的关系：

  • JDK：Java Development Kit，Java 开发工具包
  • JRE: Java Runtime Environment，Java 运行环境
  • JVM：Java Virtual Machine，Java 虚拟机
  • JDK 包含 JRE，JRE 包含 JVM

JDK 版本选择

  • 目前 JDK1.8（也叫 JDK8，注意不是 JDK18）用得最多

Java 代码的编译运行流程

  • 将 Java 源码编译成 Java 字节码。
  • 使用 JVM 将 Java 字节码转化成机器码。
  • JVM 作用：跨平台、内存管理、安全。

JSE、JEE、JME 的区别

  • JSE: Java Standard Edition，标准版
  • JEE：Java Enterprise Edition，企业版
  • JME: Java Mirco Edition，移动版
  • Spring 是 JEE 的轻量级替代品
  • SpringBoot 是 Spring + 自动化配置

JVM 是 Java 解释器，Java 运行与 c 不同，c 编译以后会变成 exe 文件，直接运行，就是一个机器码，Java 会有中间代码，叫做 Java 字节码，需要用 Java 解释器运行
JRE 就是 JVM 配上一些常用的库和包，JDK 就是在 JRE 的基础上加一些编译或者调试的工具

# 项目设计

首先项目设计，包括项目名称和项目包含模块，以及运行模 (前后端分离或前后端不分离)
前后端关联
前端发送请求，后端返回请求，网页为例，类似于函数


前后端不分离：前端向后端发送请求，后端直接向前端发送，html 代码在服务器端生成，实际上实现是从 f () 函数体内写网页文件。在这种模式下，后端通常会生成完整的 HTML 页面并将其发送给前端。这意味着 HTML 代码、CSS 样式和 JavaScript 逻辑都在服务器端生成和处理，客户端直接接收这些已经生成好的页面。后端代码如 f () 函数会直接渲染页面内容并返回。
前后端分离：前端向后端发送请求，后端先发送静态文件，前端再发送数据请求，然后后端再返回数据，前端收到数据后便静态的将数据填入页面，实际上实现是从 f () 函数内返回 html 的链接，html 文件另外书写。在这种模式中，前端和后端的职责是分开的。后端通常提供 API 接口，前端通过这些接口获取数据。前端独立处理页面的展示逻辑，包括从后端获取的数据如何呈现。前端会请求静态的 HTML 文件，或使用 JavaScript 框架动态加载和渲染页面，后端返回的数据通过 JavaScript 进行处理和填充。
总结：前后端不分离通常意味着服务器端负责生成整个页面，而前后端分离则允许前端和后端独立处理各自的任务，通过 API 进行数据交互。
两者区别可以通过数据是直接在发送过程中生成的还是后面填入的进行区分

本次项目采用前后端分离：

  • 后端 (Springboot)：Mysql,Redis, 硬盘存储，微服务
  • 前端 (web/app)：Vue3 (打包大量文件成为一个文件)

template 部分

  • <slot></slot> ：存放父组件传过来的 children 。
  • v-on:click 或 @click 属性：绑定事件
  • v-if、v-else、v-else-if 属性：判断
  • v-for 属性：循环， :key 循环的每个元素需要有唯一的 key
  • v-bind : 或 : ：绑定属性

style 部分

  • <style> 标签添加 scope 属性后，不同组件间的 css 不会相互影响。

第三方组件

  • view-router 包：实现路由功能。
  • vuex ：存储全局状态，全局唯一。
    • state : 存储所有数据，可以用 modules 属性划分成若干模块
    • getters ：根据 state 中的值计算新的值
    • mutations ：所有对 state 的修改操作都需要定义在这里，不支持异步，可以通过 $store.commit() 触发
    • actions ：定义对 state 的复杂修改操作，支持异步，可以通过 $store.dispatch() 触发。注意不能直接修改 state ，只能通过 mutations 修改 state 。
    • modules ：定义 state 的子模块

script 部分
export default 对象的属性：

  • name ：组件的名称
  • components ：存储 <template> 中用到的所有组件
  • props ：存储父组件传递给子组件的数据
  • watch() ：当某个数据发生变化时触发
  • computed ：动态计算某个数据
  • setup(props, context) ：初始化变量、函数
    • ref 定义变量，可以用 .value 属性重新赋值
    • reactive 定义对象，不可重新赋值
    • props 存储父组件传递过来的数据
    • context.emit() ：触发父组件绑定的函数

# GET、POST、PUT、DELET

@GetMapping ：处理 GET 请求，获取资源。
@PostMapping ：处理 POST 请求，新增资源。

# 配置文件

在 Spring Boot 项目中，配置文件的主要作用是管理应用程序的属性设置和外部化配置。Spring Boot 支持两种主要的配置文件格式： application.properties 和 application.yml 。这两种文件格式可以用于设置应用程序的各种配置属性，例如数据库连接、服务器端口、日志设置等。下面将详细介绍这两种配置文件的作用和使用方法。

  • application.properties 和 application.yml 都可以用于 Spring Boot 的配置，选择哪种格式主要取决于项目的需求和个人偏好。
  • 对于简单的配置， application.properties 更加直接，而对于复杂的配置结构， application.yml 更加易读和可维护。
  • 两者都可以通过注解的方式轻松读取配置。

根据项目的具体需求选择适合的配置文件格式，将大大提高项目的可维护性和可读性，实际开发使用 yml 配置文件

  • 选择方法：通常建议使用 @ConfigurationProperties ，因为它提供了类型安全的绑定和更好的可读性，尤其是在处理复杂配置时。

这种方式让应用程序的配置更加清晰，易于管理和维护。

# Bean

在 Spring 框架中，Bean 是指由 Spring IoC（控制反转）容器管理的对象。Spring 通过配置（XML、注解或 Java 配置类）来定义和创建这些对象，并负责它们的生命周期管理。下面将详细解释 Spring 中的 Bean 的相关概念。
注意在 springboot 中启动类默认会自动扫描启动类包下以及其子包

# 1. Bean 的定义

  • Bean: 在 Spring 中，Bean 是一个被 Spring 容器实例化、组装和管理的对象。Bean 可以是任何普通的 Java 对象，通常用于业务逻辑、数据访问等。

# 3. Bean 的作用域

Spring 支持多种 Bean 作用域，定义了 Bean 的生命周期和可见性：

  • singleton（默认）：在 Spring IoC 容器中只创建一个 Bean 实例，并在每次请求时返回同一实例。

  • prototype：每次请求都会创建一个新的 Bean 实例。

  • request：在 Web 应用中，每个 HTTP 请求都会创建一个新的 Bean 实例，作用域仅限于当前请求。

  • session：在 Web 应用中，每个 HTTP 会话都会创建一个新的 Bean 实例，作用域限于当前会话。

  • globalSession：在多重会话中（如 Portlet），每个全局 HTTP 会话都会创建一个新的 Bean 实例。

# 4. Bean 的生命周期

Bean 的生命周期由以下几个阶段组成：

  1. 实例化: Spring 容器创建 Bean 的实例。
  2. 属性填充: Spring 通过设置器方法或字段注入 Bean 的属性。
  3. 调用初始化方法：如果 Bean 实现了 InitializingBean 接口或配置了初始化方法，Spring 会调用它们。
  4. 使用 Bean: Bean 可以被应用程序使用。
  5. 调用销毁方法：如果 Bean 实现了 DisposableBean 接口或配置了销毁方法，Spring 会在容器关闭时调用它们。

# 5. Bean 的依赖注入

Spring 的核心特性之一是依赖注入（DI），即将对象所依赖的其他对象（依赖）注入到它的构造函数、方法或字段中。常见的依赖注入方式有：

  • 构造器注入：通过构造函数注入依赖。

  • Setter 注入：通过 setter 方法注入依赖。

  • 字段注入：直接在字段上使用 @Autowired 注解注入依赖（不推荐，因为它使得测试和依赖管理变得困难）。

# 6. Bean 的自动装配

Spring 还支持自动装配（Auto-Wiring），可以通过 @Autowired 、 @Inject 等注解自动注入依赖。常用的自动装配方式有：

  • 按类型自动装配: Spring 会根据类型查找合适的 Bean。
  • 按名称自动装配: Spring 会根据 Bean 的名称查找。
  • 构造器自动装配：通过构造函数自动装配依赖。
